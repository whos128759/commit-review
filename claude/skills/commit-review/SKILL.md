# 提交审核

只分析当前仓库 `HEAD` 已提交的内容。以 Git 证据为准；证据不足时明确写“待确认”，禁止用常见做法、相邻功能或经验补全事实。

## 收集证据

依次执行：

1. 用 `git rev-parse --verify HEAD` 确认存在提交。
2. 用 `git show -s --format=fuller HEAD` 获取作者、提交者和日期。
3. 用 `git show -s --format=%B HEAD` 获取完整提交说明，包括 `Root Cause`、`Solution`、车型和 `Change-Id`。
4. 用 `git diff-tree --no-commit-id --name-status -r -M HEAD` 获取已提交的变更文件。不要把工作区未提交改动计入影响范围。
5. 用 `git show --format= --find-renames HEAD -- .` 获取完整 diff。
6. 仅当需要确认改动符号或资源对应的直接功能入口时，搜索其引用；不要借此扩展到无关功能。

任一命令失败时停止生成审核说明，直接报告失败的命令和错误。不要根据已有片段继续猜测。

## 计算验证日期

取 `git show -s --format=%cI HEAD` 返回的提交者日期，在该日期上增加一个日历日，并格式化为 `YYYY-MM-DD`。

- 必须使用系统日期工具完成计算，禁止心算或使用当前日期。
- Windows PowerShell 可使用：`$d = git show -s --format=%cI HEAD; ([DateTimeOffset]::Parse($d).Date.AddDays(1)).ToString('yyyy-MM-dd')`。
- 如果日期为空、格式无效或无法调用日期工具，停止并报告日期计算失败；不要填写估算日期。

## 提取审核内容

### 问题原因与修改方案

- 优先使用提交说明中明确的 `Root Cause` 和 `Solution`。
- 提交说明缺失时，只能写 diff 能直接证明的事实，并注明“根据 diff 可确认”。
- 如果 diff 也不能证明，写 `待确认：提交说明和 diff 未提供足够证据。`
- 提交说明与 diff 冲突时，指出冲突，不替提交者选择一个解释。

### 影响范围

- 结合提交标题、正文、diff、改动符号及其直接引用，描述本次修改直接影响的功能，不输出变更文件清单。
- 按“车型或模块 + 功能 + 受影响内容”组织；同一功能涉及多个文件、资源或语言时合并描述。
- 只写能由本次提交直接证明的影响，不扩展间接依赖、相邻页面、其他车型或推测影响。
- 如果无法确认受影响功能，写 `待确认：无法从本次修改确认受影响功能。`

### 代码提交

- 只有本地证据中存在可直接访问的 Gerrit 变更 URL 时才填写 URL。
- 只有 `Change-Id` 而没有经过验证的 Gerrit 地址时，固定写 `（待推送 Gerrit 后补充 URL）`；禁止拼接或猜测服务器地址。

### 建议测试范围

测试范围必须直接对应本次问题和 diff 修改的功能，按以下顺序生成：

1. 从提交标题、正文、变更路径和改动符号中确认车型与问题功能。
2. 必要时搜索改动符号或资源的直接引用，确认可执行的功能入口。
3. 只描述触发本次问题所需的前置条件、操作和预期结果。
4. 仅当 diff 明确修改了分支、状态或边界值时，增加对应分支或边界测试。

禁止添加以下无直接证据的内容：

- 其他车型、相邻设置页或“全量回归”。
- 网络、蓝牙、性能、兼容性、重启、异常场景等通用测试项。
- diff 未涉及的状态组合、用户路径或预期行为。
- “建议顺便验证”“确保其他功能正常”等不可追溯表述。

如果无法从证据确认具体入口、操作或预期结果，写 `待确认：缺少<具体信息>，无法给出可执行测试步骤。`，不要生成看似合理的测试项。

### AI 评审提示（可选）

对本次 `HEAD` diff 执行宽松、低频的代码审核。检查改动在当前上下文中的语法和合理性，但只在本次修改会造成确定性高、影响严重的语法错误或编译失败时进行干预。

审核范围：

- 只检查本次提交新增或修改的代码行，以及判断这些行所必需的最小上下文。
- 不检查工作区未提交内容、未修改代码、历史问题或相邻功能。
- 合理性检查只用于确认改动是否符合当前语言语法、类型、方法签名和资源引用约束，不输出一般设计或实现偏好。

只有同时满足以下条件时才输出 `AI 评审提示`：

1. 问题由本次提交的修改直接引入。
2. 能从 diff、必要上下文或明确的编译诊断中直接证明，不依赖推测。
3. 问题会导致严重语法或解析错误、符号无法解析、类型不匹配、方法签名错误、无效资源引用等明确的编译失败。
4. 能提供针对本次修改的最小修正建议。

以下情况保持静默，不输出标题或内容：

- 本次修改未发现上述高危问题，即审核通过。
- 代码风格、命名、格式、重复、可读性、优化或重构建议。
- 一般逻辑风险、警告、Lint 提示、推测性问题或需要额外业务假设才能成立的问题。
- 与本次修改无关或修改前已经存在的问题。

发现符合条件的问题时，每项只写文件和修改行、可验证的问题依据及最小建议；不要扩展为完整代码评审。格式为：

```text
AI 评审提示：
- <文件:修改行>：<可直接证明的严重问题>。建议：<最小修正方案>。
```

### 自测结果

不要把建议测试描述成已经执行。固定输出：`锁版分支请附上自测图片或视频。`

## 输出模板

严格按以下顺序输出，不附加分析过程。除“影响范围”和“建议测试范围”外，每个字段控制在 1 至 3 句话。

```text
请在 <提交日期加 1 天，YYYY-MM-DD> 以后的版本上进行验证。

问题原因：
<证据支持的根因；证据不足则按照Root Cause内容写>

修改方案：
<证据支持的修改；证据不足则按照Solution内容写>

影响范围：
<本次修改直接影响的车型或模块、功能及具体内容；无法确认则写待确认>

代码提交：
<已验证的 Gerrit URL，或（待推送 Gerrit 后补充 URL）>

建议测试范围：
<仅列出本次问题功能的可执行测试步骤和预期结果；证据不足则写待确认>

自测结果：
锁版分支请附上自测图片或视频。
```

仅当发现符合“AI 评审提示”门槛的问题时，才在“建议测试范围”和“自测结果”之间插入该区块。审核通过时必须完全省略 `AI 评审提示` 标题及内容。
